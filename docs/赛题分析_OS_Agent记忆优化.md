## 二、基础知识科普

### 2.1 OS Agent 是什么？

运行在操作系统层面、能调用系统工具（文件系统、浏览器、邮件客户端等）的AI智能体。可以自主完成复杂任务——比如"整理上周会议纪要，提取所有人的待办事项，分别发邮件通知"，它能自己打开文件、提取内容、写邮件、发送。

### 2.2 Agent的记忆系统：三层结构

#### 短期记忆（Short-term Memory）
**当前对话的上下文**。类比：你正在跟人聊天，记得刚才说了什么。对话结束就忘。实现方式通常是对话历史存在内存里。

#### 中期记忆（Medium-term Memory）
**跨会话、但有时效性的记忆**。类比：你记得这周在做的项目细节，但上个月的项目记不太清了。通常用向量数据库存储近期交互，按时间衰减权重。

#### 长期记忆（Long-term Memory）—— 本赛题核心
**持久化、结构化的知识积累**，分两类：

- **偏好记忆（Preference Memory）**：Agent记住"你喜欢怎么做事情"。比如你总用Markdown格式、写邮件喜欢简洁风格、处理敏感数据要先脱敏。偏好不是一成不变的——会随着你的习惯变化而更新（= "动态偏好捕捉"）。

- **知识记忆（Knowledge Memory）**：Agent积累的"可复用的经验"。比如上次处理类似任务的工作流程、某类文件的标准处理模板。类比老员工的经验手册。

### 2.3 工具调用（Tool Calling）—— 记忆的数据源

Agent通过调用工具（读文件、发邮件、搜索等）获取信息。工具调用的**执行结果**是偏好和知识记忆的重要数据来源。

**问题**：工具返回的结果格式五花八门、质量参差不齐。从原始结果里直接提取偏好和知识，就像从垃圾堆里找有用信息——费力又不准。

→ 这就是要求(1)"多源数据整合"要解决的。

### 2.4 偏好提取怎么做？—— 一个例子

| 信号来源 | 原始数据 | 可提取的偏好 |
|----------|----------|--------------|
| 工具调用 | 每次都选"简洁模式"发邮件 | 输出风格偏好：简洁 |
| 手动配置 | 在设置里选了暗色主题 | 界面偏好：暗色 |
| 行为数据 | 处理代码文件时从不联网 | 安全偏好：离线优先 |

**挑战**：信号分散、偏好会变、不同场景可能冲突。

"版本化管理"：偏好不是覆盖旧的，而是像Git一样有版本记录，可以回溯到之前的状态。

### 2.5 Embedding（嵌入向量）

传统搜索是关键词匹配。**Embedding搜索**是把文字转换成高维向量（一串数字），语义相近的文字向量也相近。你说"写邮件的格式"，它能找到"email template"，即使没有匹配任何关键词。

麒麟要求调他们自己的embedding SDK，就是文字→向量的工具。

### 2.6 知识冲突与知识图谱

**知识冲突**：Agent学到两条矛盾的知识（3月用模板A、6月用模板B），需要判断B是A的更新版（时间更近），而不是两条并存。

**知识图谱 vs 向量检索**：

| | 向量检索 | 知识图谱 |
|------|----------|----------|
| 怎么做 | 知识存成向量，搜相似度 | 知识存成实体+关系，搜关联 |
| 适合 | "找类似的东西" | "这个东西和什么有关" |
| 举例 | "这个任务类似之前哪个" | "模板A → 用于 → 邮件 → 需要 → 附件格式" |

赛题的"关联检索"偏向知识图谱——不只是找相似，还要找有关联的。

### 2.7 端侧部署（Edge Deployment）

"端侧"就是**用户的本地机器**，不是云服务器。麒麟OS是桌面操作系统，Agent跑在用户电脑上，不能用云端大模型（延迟、隐私、成本），必须在本地的CPU/内存条件下运行。

硬约束：**检索响应 ≤ 500ms**——需要在索引结构和模型大小之间做trade-off。

## 三、7项技术要求逐条分析

### 要求(1)：多源数据整合模块
- 输入源：工具执行结果、用户行为数据、手动配置信息
- 需实现：数据清洗 → 格式标准化 → 质量校验
- **难度：中等**。本质是ETL pipeline，工程量大但技术成熟。

### 要求(2)：偏好记忆动态捕捉
- 目标：自动提取用户操作习惯、输出风格、安全策略等偏好
- 关键：版本化管理、跨场景适配、回溯
- **难度：较高**。涉及偏好表示学习、漂移检测、冲突消解。

### 要求(3)：知识记忆结构化整合
- 目标：新旧知识冲突处理、关联检索优化
- 存储：工作流程知识、历史案例、可复用模板
- **难度：较高**。本质是轻量级知识图谱+冲突消解+语义检索。

### 要求(4)：端侧适配
- 硬指标：检索响应延迟 ≤ 500ms
- 约束：调用银河麒麟embedding SDK，轻量化存储
- **难度：中等偏高**。精度与速度的trade-off，需实际拿到SDK测试。

### 要求(5)：安全与遗忘
- 功能：敏感信息识别与过滤 + NL指令驱动的精准遗忘
- **难度：中等**。可用规则+小模型实现，但"精准遗忘"对删除粒度有要求。

### 要求(6)：记忆流转
- 兼容短期、中期记忆的数据流转
- **难度：较低**。主要是接口设计与数据格式约定。

### 要求(7)：量化评测
- 标准化数据集 + 多个指标验证 + 完整测试报告
- **难度：中等**。需构建测试数据集或获取官方数据。

## 四、评审标准权重分析

| 维度 | 分值 | 分析 |
|------|------|------|
| **技术创新性** | 30分 | 算法创新程度，是否突破现有技术瓶颈 |
| **方案可行性** | 30分 | 架构合理性、端侧轻量化（需量化论证） |
| **性能指标** | 25分 | 偏好准确率≥85%、检索召回率≥85%、响应≤500ms、冲突正确率≥88% |
| **实用性** | 10分 | 与OS Agent产品的适配性 |
| **文档规范性** | 5分 | 报告质量 |

**关键洞察**：技术创新性+可行性共占60分，是胜负手。性能指标25分是硬门槛——达不到基本出局。复评擂台赛新增"设计新颖性(10分)+表述能力(20分)"，答辩也很重要。

### 复评（擂台赛）100分：
- 设计新颖性：10分
- 软件功能实现完整性：60分
- 应用价值：10分
- 表述能力等综合因素：20分

## 五、关键技术难点

| 难点 | 解释 | 为什么难 |
|------|------|---------|
| **偏好漂移检测** | 怎么发现用户的习惯变了？ | 区分"偶然一次不同"和"真的变了" |
| **跨场景偏好冲突** | 上班简洁、写论文详细，怎么区分场景？ | 场景边界模糊，需要自动聚类 |
| **新旧知识融合** | 新知识和旧知识矛盾时信哪个？ | 不是简单的"新覆盖旧"，可能有例外 |
| **500ms延迟约束** | 在眨眼间完成语义搜索 | 本地算力有限，索引精度和速度要平衡 |
| **精准遗忘** | "把上个月关于项目X的记录删掉" | 模糊NL指令→精确数据条目的映射 |

## 六、可行性评估（针对华科计算机大二团队）

### 优势
- 技术栈在计算机专业课程范围内可覆盖
- AI辅助开发可大幅提升编码效率
- 4个月开发周期对大二紧张但可行（每天2-3小时）

### 风险
- 团队整体技术能力一般，核心架构设计压力集中
- 麒麟embedding SDK是封闭的，需提前确认获取时间
- 评测标准缺乏明确ground truth，评测方案设计本身是难点
- 与AI+CAD研究方向不直接相关，时间投入是零和的

### 战略建议
- 20个获奖名额，参赛队伍基数通常不大，获奖概率相对较高
- 特等奖突破口在技术创新性的30分——常规方案（规则引擎+向量检索）拿不到高分
- 可考虑引入大模型驱动的偏好推理或增量式知识图谱构建
- 最大不确定性：麒麟SDK的可用性和文档质量
